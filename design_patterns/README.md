
## 设计模式的六大原则

**单一职责原则（SRP，Single Responsibility Principle）**：单一职责原则 要求一个类只负责一项职责，避免职责耦合。职责过多会导致修改一个功能时影响其他功能。例如，将用户管理和邮件通知分离为 UserService 和 EmailService 两个类，确保每个类只专注于自己的职责。

**里氏替换原则（LSP，Liskov Substitution Principle）**：里氏替换原则 要求子类能够替代父类而不影响程序的正确性。子类可以扩展父类的功能，但不能改变父类的行为。例如，子类在重写父类方法时，输入参数范围应更宽松，返回值范围应更严格。

**接口隔离原则（ISP，Interface Segregation Principle）**：接口隔离原则 主张将大而全的接口拆分为多个小接口，避免类依赖不需要的方法。例如，在智能家居系统中，灯泡只需实现开关和亮度调节接口，而空调则实现温度调节接口，确保每个类只依赖其需要的功能。

**依赖倒置原则（DIP，Dependence Inversion Principle）**：依赖倒置原则 强调高层模块不依赖于低层模块，二者都依赖于抽象。通过接口或抽象类定义依赖关系，而非具体实现。例如，通知系统通过 NotificationChannel 接口实现不同的通知方式（如邮件、短信），新增通知方式时无需修改核心逻辑。

**迪米特法则（LoD，Law of Demeter）**：迪米特法则 要求一个对象只与直接相关的对象通信，减少类之间的耦合。例如，在在线书店中，顾客通过购物车与书籍交互，而不是直接操作书籍对象，从而降低耦合度，提高代码的模块化。

**开放封闭原则（OCP，Open Close Principle）**：开闭原则 强调软件实体应该对扩展开放，对修改关闭。通过抽象类或接口定义框架，具体实现通过扩展完成，避免直接修改已有代码。例如，在计算几何图形面积时，通过定义一个 Shape 接口，新增形状时只需实现该接口，而无需修改原有代码。
